
# PHP Backend API Guide for FinanceFlow

This document outlines the API contract for a PHP backend to support the FinanceFlow Next.js frontend.

## 1. General API Principles

*   **RESTful Design**: Endpoints should follow REST principles (e.g., `GET /api/wallets`, `POST /api/wallets`).
*   **JSON Communication**: The API must accept `application/json` for request bodies and respond with `application/json`.
*   **HTTP Status Codes**: Utilize standard HTTP status codes (200, 201, 204, 400, 401, 404, 500).
*   **Error Responses**: Errors should be returned as JSON with a `message` field: `{ "message": "Error details here" }`.
*   **Base URL**: The frontend is currently configured to expect the API at `http://localhost:3001/api` (this is configurable via `process.env.NEXT_PUBLIC_API_BASE_URL`).

## 2. Authentication

*   The frontend will send a JWT (JSON Web Token) as a Bearer token in the `Authorization` header for protected routes: `Authorization: Bearer <your_jwt_token>`.
*   The PHP backend needs to:
    *   Implement login/registration endpoints.
    *   Issue JWTs on successful login.
    *   Validate JWTs to identify the user for protected endpoints.
    *   Associate all user-specific data (wallets, transactions, etc.) with a `userId` derived from the token.

## 3. API Endpoints and Data Structures

All data structures should align with those defined in the frontend's `src/lib/definitions.ts`. The `id` for entities is typically generated by the backend upon creation. `userId` is derived from the authenticated user's JWT.

---

### User & UserSettings

See `User`, `UserSettings` in `definitions.ts`.

*   **`GET /api/auth/me`**
    *   **Description**: Returns the current authenticated `User` object, including their `UserSettings`.
    *   **Response**: `User` object.
*   **`PUT /api/users/me/settings`**
    *   **Description**: Updates settings for the authenticated user.
    *   **Request Body**: `Partial<UserSettings>`
    *   **Response**: Updated `User` object.
*   **`PUT /api/users/me/profile`**
    *   **Description**: Updates the authenticated user's profile information.
    *   **Request Body**: `{ "name": "string" }` (can be extended for other profile fields)
    *   **Response**: Updated `User` object.
*   **`POST /api/users/me/change-password`**
    *   **Description**: Changes the authenticated user's password.
    *   **Request Body**: `{ "currentPassword": "string", "newPassword": "string" }`
    *   **Response**: `200 OK` with `{ "message": "Password changed successfully." }` or error.

### Main Categories

See `MainCategory` in `definitions.ts`.

*   **`GET /api/main-categories`**
    *   **Response**: `MainCategory[]`
*   **`POST /api/main-categories`**
    *   **Request Body**: `Omit<MainCategory, 'id' | 'userId'>`
    *   **Response**: `201 Created`, created `MainCategory` object.
*   **`PUT /api/main-categories/{id}`**
    *   **Request Body**: `Partial<Omit<MainCategory, 'id' | 'userId'>>`
    *   **Response**: Updated `MainCategory` object.
*   **`DELETE /api/main-categories/{id}`**
    *   **Response**: `204 No Content`. (Backend should handle deletion of associated sub-categories or re-association.)

### Sub Categories

See `SubCategory` in `definitions.ts`.

*   **`GET /api/sub-categories`**
    *   **Query Params (Optional)**: `?mainCategoryId={id}`
    *   **Response**: `SubCategory[]`
*   **`POST /api/sub-categories`**
    *   **Request Body**: `Omit<SubCategory, 'id' | 'userId'>`
    *   **Response**: `201 Created`, created `SubCategory` object.
*   **`PUT /api/sub-categories/{id}`**
    *   **Request Body**: `Partial<Omit<SubCategory, 'id' | 'userId'>>`
    *   **Response**: Updated `SubCategory` object.
*   **`DELETE /api/sub-categories/{id}`**
    *   **Response**: `204 No Content`. (Backend should handle implications for transactions/budgets using this sub-category.)

### Wallets

See `Wallet` in `definitions.ts`.

*   **`GET /api/wallets`**
    *   **Response**: `Wallet[]`
*   **`POST /api/wallets`**
    *   **Request Body**: `Omit<Wallet, 'id' | 'userId'>`
    *   **Response**: `201 Created`, created `Wallet` object.
*   **`PUT /api/wallets/{id}`**
    *   **Request Body**: `Partial<Omit<Wallet, 'id' | 'userId'>>`
    *   **Response**: Updated `Wallet` object.
*   **`DELETE /api/wallets/{id}`**
    *   **Response**: `204 No Content`. (Backend should consider how to handle transactions/transfers associated with a deleted wallet.)

### Transactions

See `Transaction` in `definitions.ts`. `createdAt` is an ISO 8601 date string.

*   **`GET /api/transactions`**
    *   **Query Params (Optional)**: `startDate`, `endDate`, `type`, `walletId`, `subCategoryId`, `searchTerm`, pagination params (`page`, `limit`).
    *   **Response**: `Transaction[]` (likely paginated).
*   **`POST /api/transactions`**
    *   **Request Body**: `Omit<Transaction, 'id' | 'userId'>`
    *   **Response**: `201 Created`, created `Transaction` object.
*   **`PUT /api/transactions/{id}`**
    *   **Request Body**: `Partial<Omit<Transaction, 'id' | 'userId'>>`
    *   **Response**: Updated `Transaction` object.
*   **`DELETE /api/transactions/{id}`**
    *   **Response**: `204 No Content`.
*   **`POST /api/transactions/{id}/stop-recurring`** (or `PUT /api/transactions/{id}/frequency`)
    *   **Description**: Updates the transaction's frequency to 'One-time'.
    *   **Response**: Updated `Transaction` object.

### Transfers

See `Transfer` in `definitions.ts`. `createdAt` is an ISO 8601 date string.

*   **`GET /api/transfers`**
    *   **Response**: `Transfer[]`.
*   **`POST /api/transfers`**
    *   **Request Body**: `Omit<Transfer, 'id' | 'userId'>`
    *   **Response**: `201 Created`, created `Transfer` object. (Backend must update balances of `fromWalletId` and `toWalletId`).
*   **`DELETE /api/transfers/{id}`**
    *   **Response**: `204 No Content`. (Backend must revert balance changes on associated wallets).

### Budgets

See `Budget` in `definitions.ts`.

*   **`GET /api/budgets`**
    *   **Query Params (Optional)**: `?month={m}&year={y}` (where `m` is 1-12).
    *   **Response**: `Budget[]`.
*   **`POST /api/budgets`**
    *   **Request Body**: `Omit<Budget, 'id' | 'userId' | 'createdAt'>`
    *   **Response**: `201 Created`, created `Budget` object.
*   **`PUT /api/budgets/{id}`**
    *   **Request Body**: `Partial<Omit<Budget, 'id' | 'userId' | 'createdAt'>>`
    *   **Response**: Updated `Budget` object.
*   **`DELETE /api/budgets/{id}`**
    *   **Response**: `204 No Content`.

---

### Shared Capital Sessions

See `SharedCapitalSession` in `definitions.ts`. `createdAt`, `updatedAt` are ISO 8601 date strings.

*   **`GET /api/shared-capital/session`**
    *   **Description**: Returns the currently active `SharedCapitalSession` for the user, or `null`.
    *   **Response**: `SharedCapitalSession | null`.
*   **`POST /api/shared-capital/session`**
    *   **Request Body**: `{ "partnerEmail": "string" }`
    *   **Description**: Creates a new session (deactivating any existing active ones for the user).
    *   **Response**: `201 Created`, created `SharedCapitalSession` object.
*   **`DELETE /api/shared-capital/session`**
    *   **Description**: Deactivates the current user's active session.
    *   **Response**: Updated (now inactive) `SharedCapitalSession` object or `204 No Content`.

---

### Feedback

See `FeedbackItem` in `definitions.ts`. `createdAt` is an ISO 8601 date string.

*   **`GET /api/feedbacks`**
    *   **Response**: `FeedbackItem[]`.
*   **`POST /api/feedbacks`**
    *   **Request Body**: `Omit<FeedbackItem, 'id' | 'userId' | 'createdAt' | 'status'>`
    *   **Response**: `201 Created`, created `FeedbackItem` object (with default `status: 'pending'`).
*   **`PUT /api/feedbacks/{id}/status`**
    *   **Request Body**: `{ "status": FeedbackStatus }` (where `FeedbackStatus` is 'pending', 'active', or 'closed')
    *   **Response**: Updated `FeedbackItem` object.

## 4. Data Types and Formatting

*   **Dates (`createdAt`, `updatedAt`, etc.)**: Frontend expects dates as ISO 8601 strings (e.g., `"2023-10-26T10:00:00.000Z"`). PHP backend should store dates (e.g., `DATETIME`, `TIMESTAMP`) and format them as ISO 8601 strings in API responses.
*   **Amounts**: Standard numeric types.
*   **Currency Codes**: Store and expect ISO 4217 currency codes (e.g., "USD", "EUR").
*   **Boolean values**: Standard `true`/`false`.

## 5. Database Considerations for PHP Backend

*   A relational database (MySQL, PostgreSQL, etc.) is recommended.
*   Each user-specific table must have a `user_id` column linked to the `users` table.
*   Establish foreign key relationships:
    *   `sub_categories.main_category_id` -> `main_categories.id`
    *   `transactions.wallet_id` -> `wallets.id`
    *   `transactions.sub_category_id` -> `sub_categories.id` (nullable)
    *   `budgets.sub_category_id` -> `sub_categories.id`
    *   `transfers.from_wallet_id` -> `wallets.id`
    *   `transfers.to_wallet_id` -> `wallets.id`
*   Define cascade or restrict rules for deletions (e.g., what happens to transactions if a wallet is deleted?).
*   Ensure appropriate indexing for performance, especially on foreign keys and frequently queried columns (like dates for transactions).

This guide provides a comprehensive starting point for developing the PHP backend API. The frontend's `src/lib/actions.ts` and `src/lib/definitions.ts` are key files to cross-reference for data structures and expected behaviors.
